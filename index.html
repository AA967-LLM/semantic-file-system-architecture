Act I · Situation

The agent needs
to find before it can fix.

Every LLM operates inside a context window — a finite, precious workspace holding its instructions, logic, and working memory. When an agent is given a codebase to work on, it needs to locate the right files. The only tool it has is the file system itself.

Task intent
"Fix the auth bug"
Linux FHS
ls, cat, grep — one directory at a time
Context window
Fills up with useless paths & metadata
Act II · Complication

The death spiral
is O(N).

In a 10,000-file repo, the agent runs thousands of ls and cat commands. Each call dumps more directory metadata into the context window. The actual task — the real instructions — get pushed out. The agent starts to hallucinate. It forgets what it was doing. Your tokens are gone.

Token leakage per search =
Σ ( Tls_calls + Tdirectory_metadata )
As repo size N grows:
tokens wasted → O(N) · unbounded
100 files → ~450 tokens wasted  20,000 files → ~95,000 tokens wasted

Act III · Answer

We skip the maze.
Teleport straight to the code.

Instead of crawling, the agent emits a natural language Intent Vector. A local FAISS index holds the entire codebase mapped into embedding space. A Cosine Similarity search finds the exact file — in constant time, regardless of repo size.

Intent: "fix auth token expiry"
↓
~15 tokens to emit the intent
FAISS · Cosine Similarity Search
↓
zero file-system traversal
auth/token_validator.py · injected
Retrieval cost =
argmax [ CosSim( IntentVector, FileEmbedding ) ]
Regardless of repo size N:
tokens consumed → O(1) · constant · always 15
100 files → 15 tokens  20,000 files → 15 tokens
